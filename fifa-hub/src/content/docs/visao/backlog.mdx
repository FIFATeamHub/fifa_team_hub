## Backlog Geral

> No contexto do FIFA Team Hub, o **Product Backlog** é uma lista dinâmica e priorizada de funcionalidades que guia a equipa para entregar valor incremental a comissões técnicas, organizadores e auditores. O backlog evolui com feedback de uso real, testes de segurança e mudanças regulatórias.

## Critérios de Aceitação de User Stories

### US-001 — Cadastrar novos utilizadores
**[↑ Voltar para US-001](#us-001)** | **Característica:** [C05 - Autenticação e gestão](#c05) | **Requisito:** RF-01

- O sistema permite que Auditor cadastre utilizadores com nome, e-mail, senha provisória e perfil
- E-mail duplicado é rejeitado com HTTP 409
- Senha é armazenada como hash bcrypt (nunca em texto plano)
- Senha provisória é exibida apenas ao Auditor que cadastrou

### US-002 — Obrigar troca de senha no primeiro acesso
**[↑ Voltar para US-002](#us-002)** | **Característica:** [C05 - Autenticação e gestão](#c05) | **Requisito:** RF-02

- Sistema bloqueia acesso ao dashboard até troca de senha
- Nova senha deve ter no mínimo 8 caracteres
- Senha provisória é invalidada imediatamente após troca
- Login com senha provisória retorna status que força redirecionamento para troca

### US-003 — Autenticar com credenciais
**[↑ Voltar para US-003](#us-003)** | **Característica:** [C05 - Autenticação e gestão](#c05) | **Requisito:** RF-03

- Credenciais inválidas retornam HTTP 401 sem revelar qual campo está errado
- Token JWT contém: user_id, role, selection_id, exp
- Token expira em 60 minutos
- Requisições pós-expiração retornam 401 e redirecionam para login

### US-004 — Redirecionar por perfil
**[↑ Voltar para US-004](#us-004)** | **Característica:** [C05 - Autenticação e gestão](#c05) | **Requisito:** RF-04

- TECHNICAL_STAFF é redirecionado para dashboard de documentos da sua seleção
- ORGANIZER é redirecionado para visão de documentação burocrática geral
- AUDITOR é redirecionado para painel de logs de auditoria

### US-005 — Controle de acesso por perfil (RBAC)
**[↑ Voltar para US-005](#us-005)** | **Característica:** [C02 - Isolamento entre seleções](#c02) | **Requisito:** RF-05

- Acesso fora do escopo retorna HTTP 403
- TECHNICAL_STAFF: acessa apenas documentos da própria seleção
- ORGANIZER: visualiza CONVOCACAO e PASSAPORTE de todas as seleções
- AUDITOR: acessa somente painel de logs, não acessa documentos
- Toda violação gera entrada no AuditLog com action=ACCESS_DENIED

### US-006 — Upload de documentos
**[↑ Voltar para US-006](#us-006)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-06

- Apenas TECHNICAL_STAFF pode fazer upload
- Tipos aceitos: CONVOCACAO, PASSAPORTE, LAUDO_MEDICO, RELATORIO_TATICO, ESQUEMA_JOGADAS
- Formatos aceitos: PDF, JPG, PNG, DOCX, máximo 10 MB
- selection_id é preenchido automaticamente pelo sistema
- Upload gera registo em AuditLog com action=UPLOAD

### US-007 — Reenvio de versões
**[↑ Voltar para US-007](#us-007)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-07

- Versão anterior é marcada como histórica (não excluída)
- Reenvio gera registo em AuditLog com action=REUPLOAD
- Apenas o autor original do documento pode reenviar
- Histórico completo é mantido para rastreabilidade

### US-008 — Listagem de documentos
**[↑ Voltar para US-008](#us-008)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-08

- Listagem exibe apenas documentos da seleção do utilizador
- Ordenação padrão: created_at DESC
- Listagem exibe: nome, tipo, responsável, data/hora, status
- ORGANIZER vê apenas CONVOCACAO e PASSAPORTE de todas as seleções
- Filtro por selection_id é aplicado no backend, nunca no frontend

### US-009 — Download com link temporário
**[↑ Voltar para US-009](#us-009)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-09

- Link temporário é válido por no máximo 15 minutos
- Link expira após o primeiro acesso (o que ocorrer primeiro)
- Download gera registo em AuditLog com action=DOWNLOAD
- Tentativa de acesso com link expirado retorna HTTP 410

### US-010 — Visualização de status
**[↑ Voltar para US-010](#us-010)** | **Característica:** [C04 - Status e conformidade](#c04) | **Requisito:** RF-10

- Status inicial de todo documento é PENDENTE
- Estados possíveis: PENDENTE, APROVADO, REJEITADO
- Mudança de status é refletida no próximo carregamento da página
- Documentos REJEITADO exibem o comentário do Organizer
- Alterações geram notificação visual ao utilizador

### US-011 — Cadastro de seleção
**[↑ Voltar para US-011](#us-011)** | **Característica:** [C05 - Autenticação e gestão](#c05) | **Requisito:** RF-11

- Organizer cadastra seleção com nome completo, código FIFA e bandeira (opcional)
- Código FIFA é único, contém 3 letras maiúsculas (ex: BRA, ARG, GER)
- Duplicata de código retorna HTTP 409
- Bandeira, se enviada, é armazenada no Cloud Storage
- Todo cadastro requer ao menos 1 utilizador Auditor vinculado

### US-012 — Isolamento inviolável
**[↑ Voltar para US-012](#us-012)** | **Característica:** [C02 - Isolamento entre seleções](#c02) | **Requisito:** RF-12

- Toda query ao banco inclui filtro obrigatório por selection_id
- Acesso via URL direta a documento de outra seleção retorna HTTP 403
- Toda tentativa de acesso indevido gera registo com action=ACCESS_DENIED
- Testes de isolamento são executados e aprovados antes de cada release
- Nenhum cenário permite cross-selection access

### US-013 — Upload de laudos médicos
**[↑ Voltar para US-013](#us-013)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-13

- Medical Staff faz upload de laudos vinculados a um atleta específico
- Tipo LAUDO_MEDICO segue as mesmas regras de upload do RF-06
- Upload registra: user_id, selection_id, nome do jogador (opcional), timestamp
- Status inicial é PENDENTE

### US-014 — Aprovação de laudos
**[↑ Voltar para US-014](#us-014)** | **Característica:** [C04 - Status e conformidade](#c04) | **Requisito:** RF-14

- Medical Staff aprova ou rejeita laudos com status PENDENTE
- Rejeição sem comentário retorna HTTP 400
- Comentário é obrigatório apenas na rejeição
- Mudança de status gera registo em AuditLog com action=APPROVE ou REJECT
- Atleta recebe notificação no próximo acesso

### US-015 — Registo de eventos
**[↑ Voltar para US-015](#us-015)** | **Característica:** [C03 - Logs imutáveis e auditoria](#c03) | **Requisito:** RF-15

- Sistema registra automaticamente: LOGIN, LOGOUT, UPLOAD, DOWNLOAD, DELETE, APPROVE, REJECT, ACCESS_DENIED
- Cada registo contém: user_id, action, resource_id, status (SUCCESS/FAILURE), ip_address, timestamp
- Registos são criados mesmo em caso de falha da ação principal
- AuditLog é imutável — nenhum perfil pode editar ou excluir

### US-016 — Visualização de logs
**[↑ Voltar para US-016](#us-016)** | **Característica:** [C03 - Logs imutáveis e auditoria](#c03) | **Requisito:** RF-16

- Auditor acessa painel de logs e visualiza histórico completo
- Filtros disponíveis: tipo de ação, utilizador, seleção, período
- Listagem exibe: utilizador, ação, recurso, data/hora, IP, resultado
- Conteúdo dos documentos nunca é exposto nos logs
- Filtros podem ser combinados para análise detalhada

### US-017 — Upload de documentos técnicos
**[↑ Voltar para US-017](#us-017)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-17

- Types RELATORIO_TATICO e ESQUEMA_JOGADAS seguem as mesmas regras de upload
- Tipos técnicos nunca aparecem na listagem do Organizer
- Acesso por outro utilizador retorna HTTP 403
- Restrição é aplicada tanto no backend quanto na apresentação

### US-018 — Armazenamento em GCS
**[↑ Voltar para US-018](#us-018)** | **Característica:** [C01 - Upload e gestão de documentos](#c01) | **Requisito:** RF-18

- Nenhum ficheiro é armazenado no servidor da aplicação
- Todos os ficheiros são armazenados no Google Cloud Storage
- URL do GCS nunca é exposta diretamente ao cliente
- Apenas links temporários assinados são fornecidos (RF-09)
- Falha no GCS durante upload retorna HTTP 503 sem persistir metadados incompletos

## Priorização Backlog (MoSCoW)

> O **método MoSCoW** classifica funcionalidades em quatro categorias para ajudar a equipa a focar no essencial para o sucesso do projeto.

### Critérios de Classificação

<CardGrid>
  <Card title="Must Have (Deve ter)" icon="check">
    Se ausente, falha o objetivo principal do projeto. A solução não entrega seu valor essencial sem estas funcionalidades.
  </Card>

  <Card title="Should Have (Deveria ter)" icon="star">
    Alto valor agregado, tempo de implementação aceitável e melhora significativa da experiência do utilizador.
  </Card>

  <Card title="Could Have (Poderia ter)" icon="plus">
    Melhorias boas de ter, mas não críticas. Implementadas se tempo e recursos permitirem.
  </Card>

  <Card title="Won't Have (Não terá agora)" icon="x">
    Fora do escopo deste MVP ou sprint. Reavaliadas em versões futuras com feedback de uso real.
  </Card>
</CardGrid>

### Pontuação de Esforço (Story Points)

A pontuação vai de **0 a 20**, sendo:

- **0–5**: Simples → Implementação em 2 dias
- **8–13**: Médio → Implementação em 5 dias
- **15–20**: Complexo → Implementação em 10 dias
