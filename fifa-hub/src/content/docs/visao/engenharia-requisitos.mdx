<div class="softex-process">
> O **Processo de Engenharia de Requisitos** está vinculado ao processo de Engenharia de Software (ESW) e abrange atividades desde a descoberta das necessidades até a validação e gestão de mudanças dos requisitos ao longo do ciclo de vida do projeto.
</div>

## Atividades de Engenharia de Requisitos

### 1. Elicitação e Descoberta

#### Técnicas de Entrevista e Questionários

<ul class="softex-list">
  <li>Realizar sessões de **entrevistas** (estruturadas ou abertas) com stakeholders chave (clientes, usuários finais) para capturar suas necessidades, dores e expectativas</li>
  <li>Utilizar **questionários** para coletar dados de um público maior de forma padronizada</li>
</ul>

#### Workshops de Requisitos e Brainstorming

<ul class="softex-list">
  <li>Conduzir **sessões colaborativas** com grupos mistos de stakeholders para discutir ideias</li>
  <li>Explorar **cenários de uso** e gerar um conjunto inicial de requisitos de forma dinâmica e interativa</li>
</ul>

### 2. Análise e Consenso

#### Priorização de Requisitos (Técnica MoSCoW)

<div class="softex-process">
> Classificar os requisitos levantados para definir o escopo e garantir o alinhamento entre stakeholders
</div>

<ul class="softex-flow">
  <li>**Must-have** (essencial) — Requisitos críticos que devem estar presentes</li>
  <li>**Should-have** (importante) — Requisitos importantes mas não críticos</li>
  <li>**Could-have** (desejável) — Requisitos que agregam valor mas podem ser adiados</li>
  <li>**Won't-have** (não terá nesta versão) — Requisitos fora do escopo atual</li>
</ul>

#### Identificação e Resolução de Conflitos

<ul class="softex-list">
  <li>Analisar os requisitos para encontrar **inconsistências, ambiguidades ou contradições** entre as solicitações dos diferentes stakeholders</li>
  <li>Facilitar **negociações** para chegar a um acordo e garantir um conjunto de requisitos coeso</li>
</ul>

### 3. Declaração de Requisitos

#### Escrita de Histórias de Usuário (User Stories)

<div class="softex-process">
> Descrever as funcionalidades sob a perspectiva do usuário final seguindo o formato:
> 
> **"Como um [tipo de usuário], eu quero [realizar uma ação] para que [eu obtenha um benefício]"**
</div>

<ul class="softex-list">
  <li>Forma ágil e focada no valor para o usuário</li>
  <li>Facilita o entendimento do propósito de cada funcionalidade</li>
</ul>


### 4. Representação de Requisitos

#### Prototipação de Interface (Wireframes e Mockups)

<ul class="softex-list">
  <li>Criar **representações visuais** de baixa ou alta fidelidade das telas do sistema</li>
  <li>Ajudar os stakeholders a **visualizar a solução** e validar o fluxo de navegação</li>
  <li>Fornecer **feedback tangível** antes do início da codificação</li>
</ul>

#### Modelagem com Diagramas (UML)

<ul class="softex-list">
  <li>Utilizar **UML** (Unified Modeling Language) para representar aspectos do sistema</li>
  <li>**Diagramas de Atividades** podem ilustrar fluxos de trabalho</li>
  <li>**Diagramas de Classes** podem definir a estrutura de dados</li>
</ul>

### 5. Verificação e Validação de Requisitos

#### Sessões de Revisão e Inspeção (Walkthroughs)

<ul class="softex-list">
  <li>Apresentar a **documentação de requisitos** para a equipe do projeto e stakeholders</li>
  <li>Análise em conjunto para encontrar **erros, omissões e ambiguidades** (Verificação)</li>
</ul>

#### Validação com Protótipos e Provas de Conceito (PoC)

<ul class="softex-list">
  <li>Apresentar um **modelo interativo** ou versão funcional mínima da solução</li>
  <li>Confirmar se os requisitos especificados atendem às **necessidades reais** no ambiente de uso (Validação)</li>
</ul>

### 6. Organização e Atualização de Requisitos

#### Controle de Versão e Gestão de Mudanças

<ul class="softex-list">
  <li>Utilizar ferramentas como **Git/GitHub Issues** para documentar e controlar versões dos requisitos</li>
  <li>Gerenciar um **processo formal** para solicitação, análise de impacto, aprovação ou rejeição de alterações</li>
</ul>
